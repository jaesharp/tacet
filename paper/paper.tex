\begin{abstract}
%-------------------------------------------------------------------------------
% ~150-200 words summarizing the key contributions
% Hook: existing tools give false confidence
% Solution: calibrated Bayesian verdicts
% Results: highlight key evaluation findings

Timing side-channel attacks remain a real threat to cryptographic implementations.
Existing detection tools output Pass or Fail without reporting when measurement
conditions make results unreliable—developers see ``pass'' and assume safety,
unaware the test lacked power to detect exploitable leaks.
%
We present \tacet, a timing side-channel detection library with calibrated three-way
verdicts: Pass, Fail, or Inconclusive (\Cref{sec:approach:decisions}). \tacet uses
Bayesian inference over the Wasserstein-1 distance, outputting posterior probabilities
$\prob{\text{leak} > \theta \given \text{data}}$ to distinguish ``no leak detected''
from ``insufficient evidence.'' The system handles autocorrelated measurements via
stream-based block bootstrap and detects degraded conditions through quality gates.
%
In our evaluation, \tacet independently detected CVE-2023-49092 (MARVIN attack) in
RustCrypto's RSA implementation (\Cref{sec:evaluation:marvin}), demonstrating practical
utility on real-world vulnerabilities. Calibration validation across 800 null-hypothesis
trials yielded 0.5\% false positive rate (4/800 failures, per-model Wilson 95\% CIs:
[0\%, 1.8\%] to [0.6\%, 4.8\%]), confirming the $\leq$5\% target. On 168 cryptographic
test cases spanning Rust, C/C++, Go, and JavaScript ecosystems, \tacet maintained
zero false positives (Wilson 95\% CI: [0\%, 2.2\%]). Designed
for CI integration, \tacet eliminates flaky tests by returning Inconclusive under poor
conditions rather than unreliable verdicts. The library is implemented in Rust with
C/C++, Go, and JavaScript bindings.

\end{abstract}

\section{Introduction}
\label{sec:intro}

Timing side-channel detection is part of validating cryptographic implementations.
Existing tools output Pass or Fail without indicating when measurement conditions
make the result unreliable. When noise or autocorrelation degrades data quality,
these tools may output ``pass'' even when the evidence is insufficient—developers
see a passing result and assume their code has no exploitable leak, unaware the
test could not actually detect anything.

Timing side-channels remain a real threat. The MARVIN attack~\cite{marvin} showed
that RSA PKCS\#1 v1.5 implementations in over 40 libraries leak plaintext through
nanosecond-scale timing differences in big-integer arithmetic. Attacks span threat
models from remote network adversaries to co-located processes, with precision
requirements ranging from microseconds to individual clock cycles (\Cref{sec:background:attacks}).

The standard approach compares execution time distributions between fixed and
random inputs. \dudect~\cite{dudect}, the most widely used tool, applies Welch's
t-test. More recent tools like \rtlf~\cite{rtlf} and \silent~\cite{silent} use
quantile comparisons with bootstrap-based inference. However, these tools provide
Pass/Fail verdicts without quantifying when evidence is insufficient~\cite{trailofbits-dudect}.
Under noisy conditions—common in CI with shared resources and variable load—tools
may output ``pass'' when the evidence is genuinely insufficient, giving unwarranted
confidence in vulnerable code.

We present \tacet, a timing side-channel detection tool with three possible
verdicts: Pass, Fail, or Inconclusive (\Cref{sec:approach:decisions}). \tacet
computes the posterior probability $\prob{\text{leak} > \theta \given \text{data}}$
through Bayesian inference and only emits Pass or Fail when this probability
crosses a decision threshold (5\% or 95\%). When evidence is insufficient,
quality gates (\Cref{sec:approach:gates}) return Inconclusive.

\silent~\cite{silent} introduced an important insight: timing distributions should
be compared via quantiles rather than means to detect both uniform shifts (from branches)
and tail effects (from cache behavior). However, quantile-based methods have limited
power for rare tail effects—when leaks affect only a small fraction of samples, the signal
can dilute across multiple dimensions. We build on \silent's insight but use the
Wasserstein-1 (W$_1$) distance, a single metric that shows higher detection power for
rare tail effects in our evaluation (\Cref{sec:approach:wasserstein}). The W$_1$ distance
measures how far probability mass must move to transform one distribution into another,
so even rare outliers contribute visibly
to the statistic.

Beyond the W$_1$ statistic (\Cref{sec:approach:wasserstein}), we contribute a Bayesian
framework (\Cref{sec:approach:inference}) with calibrated three-way verdicts
(Pass/Fail/Inconclusive; \Cref{sec:approach:decisions}), quality gates that detect
degraded measurement conditions (\Cref{sec:approach:gates}), and an explicit measurement
floor (\Cref{sec:approach:floor}) that prevents claiming ``no leak'' when the system
cannot resolve effects at the requested threshold.

\section{Background and Related Work}
\label{sec:background}

%-------------------------------------------------------------------------------
\subsection{Timing Side-Channel Attacks}
\label{sec:background:attacks}

Timing side-channels exist when a cryptographic implementation's execution time
depends on secret data. An attacker who can measure this timing (either directly
or through statistical analysis of many queries) may recover secret keys or
plaintext. Attacker capabilities span three regimes: \emph{remote network}
($\sim$\us{50} precision), \emph{adjacent network} ($\sim$\ns{100}, including
HTTP/2 multiplexing~\cite{timeless}), and \emph{co-located} (cycle-level via
cache timing~\cite{kyberslash}). Attacks like Raccoon~\cite{raccoon} and
MARVIN~\cite{marvin} demonstrate that microsecond-to-nanosecond leaks enable
practical key recovery.

%-------------------------------------------------------------------------------
\subsection{Existing Detection Tools}
\label{sec:background:tools}

Several tools exist for detecting timing side-channels during development.

\paragraph{\dudect~\cite{dudect}.}
The most widely used tool. \dudect applies Welch's t-test to compare timing
distributions between fixed and random inputs. It is simple, fast, and available
in C with Rust bindings. It uses parametric statistics, prioritizing simplicity
and interpretability, and outputs a t-statistic for binary decision-making.

\paragraph{\tlsfuzzer~\cite{tlsfuzzer}.}
A network-level timing analysis tool that applies multiple statistical tests
(Wilcoxon, Friedman, etc.) to TLS implementations. It mitigates autocorrelation
via paired sampling and provides bounded Type-I error, but requires complex setup
for protocol fuzzing and outputs binary verdicts without uncertainty quantification.

\paragraph{\rtlf~\cite{rtlf}.}
Introduced at USENIX Security 2024, \rtlf compares nine deciles between timing
distributions using an i.i.d.\ bootstrap with 10,000 iterations. This captures
distributional differences beyond means. The tool is designed for independent
samples and operates on pre-collected measurement files.

\paragraph{\silent~\cite{silent}.}
The most statistically sophisticated prior work. \silent uses a 9-quantile test
statistic with $m$-dependent block bootstrap that handles autocorrelated data.
It provides bounded Type-I error via relevance testing against a user-defined
threshold $\Delta$. The approach emphasizes distributional shape via quantile
comparisons.

\section{Tacet's Approach}
\label{sec:approach}

\tacet is designed around one principle: when measurement conditions cannot support
a confident conclusion, say so rather than guess. This section describes the
statistical methodology.

\tacet operates in two phases. During \emph{initialization}, it collects an initial
sample (default 5,000 per class) to estimate the covariance structure via block
bootstrap and select the Bayesian prior scale. During \emph{adaptive sampling}, it
collects batches (default 1,000 samples) and updates the posterior probability
after each batch, continuing until a decision threshold is crossed or a quality
gate triggers Inconclusive.

\paragraph{Threshold selection.}
The threshold $\theta$ defines the minimum W$_1$ distance worth detecting. There is no
single correct value—it depends on the threat model. We provide presets derived
from published attacks:
%
\textsc{SharedHardware} (\ns{0.4}, $\approx$2 cycles at 5\,GHz) is derived from
minimum exploitable SGX cache-timing differences~\cite{sgxtiming};
%
\textsc{PostQuantumSentinel} (\ns{2.0}, $\approx$10 cycles at 5\,GHz) is derived
from KyberSlash attacks requiring $\sim$20 cycle resolution~\cite{kyberslash};
%
\textsc{AdjacentNetwork} (\ns{100}) is derived from HTTP/2 multiplexing precision
in Timeless Timing Attacks~\cite{timeless};
%
\textsc{RemoteNetwork} (\us{50}) is conservative for general internet
jitter~\cite{crosby2009}.
Users can specify custom thresholds.

%-------------------------------------------------------------------------------
\subsection{Test Statistic: Wasserstein-1 Distance}
\label{sec:approach:wasserstein}

Rather than comparing means or individual quantiles, we measure the \textbf{Wasserstein-1 (W$_1$)
distance}~\cite{villani2009}---also known as the \textbf{earth-mover distance}---between timing distributions.

\paragraph{Intuition.}
Imagine timing measurements as piles of earth arranged along a number line. W$_1$ measures
the minimum total work (amount $\times$ distance) needed to rearrange one pile to match
the other. For timing analysis, this captures ``how much timing mass must be moved, and how far.''

\paragraph{Formal definition.}
For sorted samples $f_{(i)}$ (fixed input) and $r_{(i)}$ (random input):
\begin{equation}
    \widehat{W}_1 = \frac{1}{n}\sum_{i=1}^{n} \left| r_{(i)} - f_{(i)} \right|
\end{equation}

This is the \textbf{average absolute rank-wise mismatch} between distributions, measured
in nanoseconds. We use $\widehat{W}_1$ directly as the observed statistic for Bayesian
inference without debiasing or clamping.

The empirical $\widehat{W}_1$ estimator exhibits upward bias for finite samples under the
null hypothesis. We do not apply bias correction or clamping (e.g., $\max(0, \widehat{W}_1 - c)$)
because such transformations create a point mass at zero, violating the continuous likelihood
model. The Student-$t$ likelihood approximates the sampling distribution of $\widehat{W}_1$
under the finite-sample regime (see \Cref{app:methodology}); finite-sample bias is
absorbed by the measurement floor mechanism (\Cref{sec:approach:floor}). The effective
threshold $\theta_{\text{eff}}$ rises to accommodate the smallest reliably detectable effect,
ensuring the posterior remains well-calibrated even when $\widehat{W}_1 > 0$ under the null.

\paragraph{Why W$_1$ for timing leaks?}
Timing leaks manifest in different patterns:
\begin{itemize}
    \item \textbf{Uniform shifts} (branch-based): All quantiles shift by $c$ $\Rightarrow$ $W_1 \approx |c|$
    \item \textbf{Tail effects} (cache-based): Upper quantiles shift more $\Rightarrow$ $W_1$ emphasizes tail transport
\end{itemize}

For rare tail effects (cache misses affecting a small fraction of measurements),
W$_1$ shows higher detection power than quantile-based approaches in our evaluation:
when effects were concentrated in 5\% of samples, \silent's detection power dropped
to zero (\Cref{sec:evaluation:synthetic}). Because W$_1$ measures the cost to
transport probability mass, even rare outliers contribute proportionally to their
distance from the baseline distribution—without needing to affect multiple quantiles
or requiring corrections for multiple comparisons.

%-------------------------------------------------------------------------------
\subsection{Bayesian Inference}
\label{sec:approach:inference}

The W$_1$ distance provides a scalar summary of distributional difference. We use
Bayesian inference to compute posterior probabilities instead of p-values.

\paragraph{Interpretable output.}
When \tacet reports ``72\% probability of a leak,'' this means what it says:
given the observed data and our model, 72\% of the posterior probability mass
corresponds to effects exceeding the threshold. A p-value answers a different
question (``if there were no leak, how likely is data this extreme?'') and is
routinely misinterpreted.

\paragraph{Valid adaptive sampling.}
Frequentist methods suffer from optional stopping: continuing to sample until
reaching significance inflates false positive rates. Bayesian posterior updating
is less susceptible to these inflated error rates of sequential testing, provided
the generative model is well-specified. Since \tacet fixes the prior and likelihood
during initialization before adaptive sampling begins, the posterior remains coherent
under optional stopping. This enables adaptive sampling: collect batches until
the decision threshold is crossed or a quality gate triggers.

\paragraph{Principled use of prior information.}
The prior encodes what we know before seeing data. \tacet calibrates the prior
to express genuine uncertainty: neither strongly favoring ``leak'' nor ``no leak.''
This prevents the prior from dominating when data is limited while allowing
confident conclusions when evidence is strong.

The full model specification is in \Cref{app:methodology}. At a high
level: we model the observed W$_1$ distance with a robust Student-$t$ likelihood
that automatically inflates uncertainty when the variance estimate is too small,
and place a Half-Student-$t$ prior on the true W$_1$ distance with scale selected
based on the user-requested threshold. We approximate the posterior via Gibbs sampling
(256 iterations) with $<$\ms{1} computational overhead.

%-------------------------------------------------------------------------------
\subsection{Calibrated Verdicts}
\label{sec:approach:decisions}

The posterior probability $\prob{\text{leak} > \theta \given \text{data}}$ maps to
three verdicts. Unlike binary pass/fail tools, \tacet can report when evidence is
insufficient.

We say verdicts are \emph{calibrated} if, under the null hypothesis
(no timing leak exceeding $\theta$), $P(\text{Fail}) \leq \alpha$,
and under alternatives exceeding $\theta$, $P(\text{Pass}) \leq \beta$,
where $\alpha$ and $\beta$ are the decision thresholds (default 5\%).
We use these posterior thresholds as a design choice intended to achieve
the stated frequentist bounds; \Cref{sec:evaluation:results} validates
empirically that this holds across our benchmark conditions.

\begin{itemize}
    \item \textbf{Pass}: The posterior probability of a timing leak exceeding
          the effective threshold is below 5\%: $\prob{\text{leak} > \theta_{\text{eff}} \given \text{data}} < 0.05$
          (\Cref{sec:approach:floor}).
          High confidence that no timing leak exceeding $\theta_{\text{eff}}$ exists under the tested
          input distribution.

    \item \textbf{Fail}: The posterior probability exceeds 95\%:
          $\prob{\text{leak} > \theta_{\text{eff}} \given \text{data}} > 0.95$.
          High confidence that a timing leak exists.

    \item \textbf{Inconclusive}: The posterior probability falls between the
          decision thresholds, \emph{or} a quality gate has detected that
          measurement conditions cannot support a reliable verdict.
\end{itemize}

Binary tools cannot distinguish between ``no leak above threshold'' and
``insufficient evidence.'' \tacet makes this explicit: Inconclusive means
the test should be re-run under better conditions or with more samples,
not that the implementation is safe.

For CI integration, this prevents flaky tests. Rather than randomly passing or
failing based on environmental noise, \tacet returns Inconclusive when conditions
are poor. Developers can configure how to handle this (retry, warn, or fail-closed)
based on their security requirements.

%-------------------------------------------------------------------------------
\subsection{Handling Autocorrelation}
\label{sec:approach:autocorrelation}

Timing measurements exhibit autocorrelation: nearby samples are more similar than
distant ones due to cache state, CPU frequency scaling, and OS scheduling. Standard
statistical methods assume independent samples. Violating this assumption causes
underestimated variance and inflated false positive rates.

\tacet addresses this through block bootstrap on the acquisition stream—the
interleaved sequence of (class, timing) pairs in measurement order. Rather than
resampling individual measurements (which destroys the dependence structure),
we resample contiguous blocks and reconstruct timing distributions from each
bootstrap replicate. This preserves temporal dependence within blocks while
estimating the variance inflation caused by autocorrelation.

\tacet selects block length automatically using the Politis-White algorithm~\cite{politiswhite},
which estimates the optimal block size from the autocorrelation structure. We
also compute the integrated autocorrelation time (IACT), which quantifies how
many nominal samples equal one independent observation. The effective sample size
$n_{\text{eff}} = n / \hat{\tau}$ can be much smaller than the nominal count when
autocorrelation is strong.

\silent also uses block bootstrap to handle autocorrelation. The key distinction
is that \tacet uses \emph{stream-based bootstrap}—resampling the interleaved
acquisition stream rather than per-class samples. This preserves cross-class
temporal dependencies (e.g., cache state carrying from a Fixed measurement to
the next Random measurement). Additionally, \tacet actively monitors for
stationarity violations via quality gates, and outputs Inconclusive when
autocorrelation is too severe to handle, rather than forcing a verdict.

%-------------------------------------------------------------------------------
\subsection{Quality Gates}
\label{sec:approach:gates}

Quality gates enable calibrated verdicts by detecting when measurement conditions
cannot support a confident conclusion. We use a two-track numbering scheme to
distinguish categories: \textbf{S1--S2} (statistical validity gates) detect
violations of the model's assumptions, while \textbf{O1--O2} (operational budget
policies) impose practical limits on resource consumption.

\paragraph{Statistical Validity Gates.}

\paragraph{S1: Insufficient information gain.}
After initialization, \tacet computes the KL divergence between the prior and
posterior distributions. If KL $<$ 0.7 nats, the posterior has barely moved
from the prior—the data is too noisy to distinguish signal from noise.
The 0.7 nats threshold was determined via validation experiments on synthetic
data with known ground truth: values below this indicate the posterior has
moved less than one bit of information from the prior.
This triggers Inconclusive with reason \texttt{DataTooNoisy}.

During adaptive sampling, if the KL divergence between successive posteriors
collapses (total $<$ 0.001 nats over 5+ batches), the model has stopped learning
from new data. The 0.001 nats threshold over 5 batches indicates the model has
effectively stopped learning from new data ($<$0.001 bits per batch); further
sampling is unlikely to resolve the decision. This triggers Inconclusive with
reason \texttt{NotLearning}.

\paragraph{S2: Condition drift detected.}
The covariance estimate from initialization assumes stationary conditions. If
measurement statistics change during the adaptive loop (variance ratio outside
$[0.5, 2.0]$, lag-1 autocorrelation change $>0.3$, or mean drift $>3\sigma$),
this indicates non-stationarity. The $[0.5, 2.0]$ bounds detect $\geq$2$\times$
changes in noise level, validated on synthetic non-stationary benchmarks where
measurement variance drifts over time. \tacet reports Inconclusive with reason
\texttt{ConditionsChanged} rather than producing a verdict based on invalid
assumptions.

\paragraph{Operational Budget Policies.}

\paragraph{O1: Would take too long.}
\tacet extrapolates time to decision based on the current convergence rate. If
projected time exceeds 10$\times$ the configured budget, it stops early with
Inconclusive rather than wasting resources on a test that cannot conclude within
practical limits. The 10$\times$ budget projection is an operational policy, not
a statistical threshold: it prevents runaway tests while allowing natural variance
in convergence rates.

\paragraph{O2: Resource budget exceeded.}
If elapsed time exceeds the configured time budget or samples exceed the maximum,
\tacet reports Inconclusive with the current posterior probability, allowing the
user to interpret the partial result.

%-------------------------------------------------------------------------------
\subsection{Measurement Floor}
\label{sec:approach:floor}

A subtle failure mode: claiming ``no leak'' when the measurement system cannot
actually resolve effects at the requested threshold. \tacet makes this explicit
through the measurement floor $\theta_{\text{floor}}$.

The measurement floor is the smallest effect the system can reliably detect given
current conditions. It has two components: (1) a statistical component that shrinks
as $\theta_{\text{floor,stat}}(n) = c_{\text{floor}} / \sqrt{n_{\text{blocks}}}$,
where $c_{\text{floor}}$ is calibrated from the noise structure and $n_{\text{blocks}} = \max(1, \lfloor n/L \rfloor)$; and (2) a timer
resolution component $\theta_{\text{tick}}$ based on quantization. These combine as:
\begin{equation}
    \theta_{\text{floor}} = \max(\theta_{\text{floor,stat}}(n), \theta_{\text{tick}})
\end{equation}

The effective threshold used for inference is:
\begin{equation}
    \theta_{\text{eff}} = \max(\theta_{\text{user}}, \theta_{\text{floor}})
\end{equation}
When $\theta_{\text{eff}} > \theta_{\text{user}}$, Fail still propagates (detecting
an effect above $\theta_{\text{eff}}$ implies one above $\theta_{\text{user}}$),
but Pass does not (cannot rule out effects in the gap). \tacet returns Inconclusive
with reason \texttt{ThresholdElevated}. This prevents claiming ``safe at \ns{100}''
when the system can only resolve \ns{500} effects. See \Cref{app:floor} for
complete derivation.

\section{Implementation}
\label{sec:implementation}

\tacet is implemented in Rust. The statistical core (adaptive sampling, Bayesian
inference, bootstrap resampling) is approximately 6,000 lines of code. The
bootstrap resampling is parallelized across CPU cores, providing 4--8$\times$
speedup on multi-core systems. Foreign function interfaces expose the same API
to C, C++, Go, and JavaScript (via WebAssembly with native timer bindings for
Node.js, Deno, and Bun).

%-------------------------------------------------------------------------------
\subsection{Measurement Infrastructure}
\label{sec:implementation:measurement}

High-resolution timing is platform-specific. On x86\_64, we use the \texttt{rdtsc}
instruction (cycle-level resolution). On ARM64, we use the \texttt{cntvct\_el0}
counter (coarser resolution: typically 24\,MHz on Apple Silicon, $\sim$\ns{42} ticks).

For cycle-level precision on ARM64, \tacet optionally supports Performance Monitoring
Unit (PMU) counters: \texttt{kperf} on macOS (requires elevated privileges) and
\texttt{perf\_event} on Linux (requires \texttt{CAP\_PERFMON} or root). These provide
sub-nanosecond resolution but are opt-in due to privilege requirements.

\tacet collects measurements as an interleaved acquisition stream, alternating
fixed and random inputs to minimize systematic bias from CPU state changes. The
system pre-generates all inputs before the measurement loop to avoid timing
contamination from random number generation.

%-------------------------------------------------------------------------------
\subsection{Adaptive Batching}
\label{sec:implementation:batching}

On platforms with coarse timer resolution (e.g., Apple Silicon's \ns{42}
\texttt{cntvct\_el0}), \tacet automatically batches operations to improve
effective resolution (\Cref{app:batching}).

%-------------------------------------------------------------------------------
\subsection{CI Integration}
\label{sec:implementation:bindings}

\tacet is designed for CI pipelines:

\begin{itemize}
    \item \textbf{Non-flaky}: Tests return Inconclusive under poor conditions
          rather than randomly passing or failing. This eliminates the ``rerun
          until green'' problem.

    \item \textbf{Configurable budgets}: Time and sample budgets can be set per
          test, allowing quick checks during development (10\,s) and thorough
          validation before release (60\,s+).

    \item \textbf{Exit codes}: The CLI returns distinct exit codes for Pass (0),
          Fail (1), and Inconclusive (2). Organizations can configure fail-closed
          (treat Inconclusive as failure) or fail-open (treat as warning) policies.

    \item \textbf{Structured output}: JSON output includes the posterior probability,
          effect size estimate with credible intervals, and diagnostics.
\end{itemize}

\subsection{Methodology}
\label{sec:evaluation:methodology}

\paragraph{Synthetic benchmarks.}
Synthetic experiments use generated timing distributions with controlled effect
sizes, autocorrelation structures, and noise levels. This allows precise ground-truth
evaluation independent of hardware. We inject effects as shifts in mean timing
(simulating branch differences) or shifts in upper quantiles only (simulating
cache effects), with autocorrelation modeled as AR(1) processes at varying
correlation coefficients $\rho \in \{-0.6, -0.3, 0, 0.3, 0.6\}$.

\paragraph{Cryptographic library tests.}
Real-world tests run on AWS EC2 instances: \texttt{c8g.4xlarge} (ARM64 Graviton4,
16 vCPUs) and \texttt{c8a.4xlarge} (x86\_64 AMD EPYC, 16 vCPUs)—typical CI
infrastructure. We test cryptographic implementations across four ecosystems:
Rust (RustCrypto, ring, dalek, pqcrypto, orion), C/C++ via FFI (LibreSSL, Libsodium,
wolfSSL, mbedTLS, Botan), Go standard library, and JavaScript/WASM (node-forge,
crypto-js, Noble cryptography). Coverage spans symmetric ciphers, asymmetric crypto,
hashes, AEAD, and post-quantum algorithms.\footnote{BoringSSL tests were developed
but cannot be integrated due to Rust linker conflicts between \texttt{openssl}
and \texttt{boring} crates. Go's circl library (post-quantum) tests are blocked
by an FFI bug in tacet-go bindings.}

\paragraph{Tools compared.}
We compare \tacet against \dudect, \silent, \rtlf, and several additional
statistical tests (Kolmogorov-Smirnov, Anderson-Darling, TVLA t-test, MONA).
For fair comparison, all tools receive identical sample counts (10,000 samples
per class for synthetic benchmarks, matching \dudect's minimum reporting
threshold~\cite{dudect}).

\paragraph{Metrics.}
We report: (1) \emph{False positive rate} (FPR): proportion of null-effect
trials yielding Fail; (2) \emph{Detection rate}: proportion of effect-present
trials yielding Fail; (3) \emph{Inconclusive rate}: proportion of trials yielding
Inconclusive (applicable only to \tacet). For tools without an Inconclusive
category, all verdicts are either Pass or Fail.

\subsection{Case Study: Blind Re-Detection of MARVIN}
\label{sec:evaluation:marvin}

When testing RustCrypto's \texttt{rsa} crate v0.9.9—documented as constant-time
in the library changelog—\tacet detected a timing difference in RSA-1024 decryption
that exceeded the \textsc{AdjacentNetwork} threshold (\ns{100}).

\paragraph{Detection details.}
The oracle collected 50,000 samples over 30 seconds, achieving
$\prob{\text{leak} > \ns{100} \given \text{data}} > 0.95$ (Fail verdict).
Effect estimate: \ns{250} (95\% credible interval: [180, 320]~ns). The leak
appeared consistently across multiple test runs with different random seeds.

\paragraph{Post-hoc identification.}
This vulnerability was CVE-2023-49092 (MARVIN attack)~\cite{marvin}, which we
were unaware of when adding RSA to our test corpus. The leak originates from
data-dependent branching in big-integer operations during PKCS\#1 v1.5
padding verification. The attack affects over 40 cryptographic libraries and enables
plaintext recovery through timing oracle queries. This blind detection validates
\tacet's practical utility for finding real vulnerabilities.

\paragraph{Accepted timing leaks.}
\tacet also detected timing leaks that turned out to be documented and accepted by
implementers. Testing noble-ciphers~\cite{noble} (JavaScript) without prior
knowledge of its implementation choices, \tacet detected AES-256-CTR key expansion
timing ($P(\text{leak})=0.96$, 45--\ns{120} effect). Investigation revealed the library
explicitly uses T-tables~\cite{bernstein2005cache} for performance—the documentation
acknowledges this creates cache-timing leaks. Similarly, Go's \texttt{crypto/rsa}
PKCS\#1 v1.5~\cite{go-rsa-docs} showed timing variation which the documentation warns
is fundamentally unsafe for non-session-key use. Unlike the MARVIN case where detection
revealed a bug, these detections correctly identified leaks that implementers knowingly
accepted as trade-offs.

\section{Discussion and Limitations}
\label{sec:discussion}

%-------------------------------------------------------------------------------
\subsection{Tradeoffs}
\label{sec:discussion:tradeoffs}

\paragraph{Methodological complexity.}
\tacet's statistical methodology is more complex than a t-test. The full specification
(provided as an artifact) runs to approximately 30 pages, covering the Bayesian
model, bootstrap procedures, quality gates, and measurement floor calculations.
This complexity is the price of calibration: simpler methods cannot provide the
same guarantees about when results are reliable.

For practitioners, this complexity is encapsulated in the library—users do not
need to understand the statistical details. The API is simple: specify a threshold
(or attacker model), run the test, interpret the verdict.

\paragraph{Lower power under noise.}
\tacet is deliberately conservative. When measurement conditions are marginal,
it returns Inconclusive rather than committing to Pass or Fail. This may require
more samples or better conditions to reach a definitive verdict compared to tools
that always produce binary output.

\tacet trades raw detection rate for calibrated confidence: the Inconclusive
rate reflects genuine uncertainty rather than concealing it behind a binary
verdict. A tool that reports Pass under noisy conditions is not more powerful;
it is overconfident.

%-------------------------------------------------------------------------------
\subsection{Limitations}
\label{sec:discussion:limitations}

\paragraph{Threshold mis-specification sensitivity.}
The Bayesian prior is scaled based on $\theta_{\text{user}}$. If the true exploitable
effect differs significantly (e.g., user sets \ns{100} but real threats are at \ns{10}),
detection power may be reduced. Users should select thresholds conservatively based
on their threat model (\Cref{sec:approach}).

\paragraph{Stationarity violations in practice.}
While quality gates detect gross non-stationarity (gate S2), subtle drift may go
undetected. For example, gradual CPU thermal throttling may slowly increase variance
without triggering the 2$\times$ threshold. Long tests should periodically restart
to refresh initialization.

\paragraph{Model specification.}
The Bayesian model makes specific choices (Student-$t$ distributions, W$_1$ distance,
fixed degrees of freedom) validated on our benchmark suite. Different application
domains may benefit from alternative specifications. The modular design allows
researchers to experiment with variants.

%-------------------------------------------------------------------------------
\subsection{Future Work}
\label{sec:discussion:future}

\tacet includes experimental support for power and electromagnetic side-channel
analysis, using the same statistical methodology with different measurement
infrastructure (oscilloscope traces rather than CPU timers). The feature
extraction and alignment challenges differ, but the core inference framework
applies directly. We plan to validate this capability against standard SCA
benchmarks in future work.

\section{Conclusion}
\label{sec:conclusion}

\tacet independently detected CVE-2023-49092 in production RSA code
(\Cref{sec:evaluation:marvin}), demonstrating practical utility on real-world
vulnerabilities. Calibration validation across 800 null-hypothesis trials confirmed
0.5\% false positive rate (4/800 failures), consistent with the $\leq$5\% target.
Across 168 cryptographic test cases spanning four ecosystems, \tacet maintained
zero false positives (Wilson 95\% CI: [0\%, 2.2\%]) under varying measurement conditions,
while existing tools showed elevated false positive rates under autocorrelation or missed
tail-only effects.

The key methodological contributions are: Wasserstein-1 distance for tail-sensitive
detection of timing leaks; Bayesian posterior probabilities for interpretable
uncertainty quantification; stream-based block bootstrap for handling autocorrelated
data; quality gates that detect measurement validity violations and trigger
Inconclusive verdicts; and measurement floor tracking that prevents claims of
``no leak'' when the system cannot resolve effects at the requested threshold.

\tacet is designed for CI pipelines: the three-way verdict system eliminates
flaky tests by returning Inconclusive rather than randomly passing or failing
under variable conditions. Available as open-source software under the MPL~v2
license with native support for Rust, C, C++, Go, and JavaScript, \tacet
provides uncertainty-aware verdicts that align testing outcomes with measurable
evidence quality.

\section{Statistical Methodology Details}
\label{app:methodology}

This appendix provides additional detail on tacet's statistical methodology for reviewers familiar with Bayesian inference. We cover the probabilistic model, inference procedure, autocorrelation handling, quality gates, and measurement floor derivation. For complete algorithmic specifications, see the specification document in the artifact repository.

\subsection{Probabilistic Model}
\label{app:model}

\paragraph{Test Statistic.}
Rather than comparing means or individual quantiles, we measure the \textbf{Wasserstein-1 (W$_1$)
distance}---also known as the \textbf{earth-mover distance}---between timing distributions.

The W$_1$ distance is the minimum total cost to transform one distribution into another, where
cost is the amount of probability mass times the distance it must be moved. For discrete
empirical distributions with $n$ samples each, this reduces to:
\begin{equation}
\widehat{W}_1(F, R) = \frac{1}{n}\sum_{i=1}^{n} \left| r_{(i)} - f_{(i)} \right|
\end{equation}
where $f_{(i)}$ and $r_{(i)}$ are the sorted (order-statistic) samples from the fixed and
random input classes.

This is the \textbf{average absolute rank-wise mismatch} between distributions, measured in
nanoseconds. We use $\widehat{W}_1$ directly as the observed statistic for inference---no
debiasing or clamping is applied before computing the posterior.

\paragraph{Interpretation.}
For a uniform shift of $c$ nanoseconds, $W_1 \approx |c|$. For tail effects (e.g., cache
misses affecting upper percentiles), $W_1$ measures the total transport cost, naturally
emphasizing the distance those outliers must move. A timing leak is declared when
$W_1 > \theta$ with high posterior probability.

\paragraph{Likelihood.}
We model the observed W$_1$ distance with a robust Student-$t$ likelihood to handle potential
variance underestimation. Using the scale-mixture representation:
\begin{align}
\widehat{W}_1 \mid W, \kappa &\sim \mathcal{N}\!\left(W, \frac{v_n}{\kappa}\right) \\
\kappa &\sim \text{Gamma}\!\left(\frac{\nu_\ell}{2}, \frac{\nu_\ell}{2}\right)
\end{align}
with $\nu_\ell = 8$. Marginally, this yields $\widehat{W}_1 \mid W \sim t_{\nu_\ell}(W, v_n)$.

When the estimated variance $v_n$ is too small (e.g., from unmodeled autocorrelation), the
scale parameter $\kappa$ is pulled below 1, automatically inflating uncertainty. This prevents
overconfident posteriors when our dependence model is mis-specified.

The variance $v_n$ is computed from calibration-phase bootstrap as:
\begin{equation}
v_n = \frac{v_{\text{rate}}}{n_{\text{blocks}}(n)}
\end{equation}
where $v_{\text{rate}}$ is the variance-rate constant estimated during calibration, and
$n_{\text{blocks}}(n) = \max(1, \lfloor n/L \rfloor)$ is the number of approximately independent
blocks given block length $L$.

\paragraph{Prior.}
We place a Half-Student-$t$ prior on the true W$_1$ distance $W$:
\begin{equation}
W \sim \text{Half-}t_\nu(0, \sigma_t^2), \quad \nu = 4
\end{equation}
where the prior scale $\sigma_t$ is calibrated so that the prior probability of exceeding
the \textbf{user-requested threshold} $\theta_{\text{user}}$ equals a target probability
$\pi_0 = 0.62$:
\begin{equation}
P\!\left(W > \theta_{\text{user}} \mid W \sim \text{Half-}t_\nu(0, \sigma_t^2)\right) = \pi_0
\end{equation}

This calibration targets the \emph{threat-model threshold} (what the user wants to detect),
not the measurement floor. The value $\pi_0 = 0.62$ expresses genuine prior uncertainty
without strong bias toward either ``leak'' or ``no leak.'' It is chosen to be slightly above
0.5 to account for the Half-t distribution's support being restricted to positive values.

We implement inference using the equivalent scale-mixture representation:
\begin{align}
\lambda &\sim \text{Gamma}\!\left(\frac{\nu}{2}, \frac{\nu}{2}\right) \\
W \mid \lambda &\sim \text{Half-}\mathcal{N}\!\left(0, \frac{\sigma_t^2}{\lambda}\right)
\end{align}
which enables closed-form Gibbs conditionals.

\subsection{Inference Procedure}
\label{app:inference}

\paragraph{Gibbs Sampler.}
We approximate the posterior $p(W \mid \widehat{W}_1)$ using a Gibbs sampler over $(W, \lambda, \kappa)$.
Each iteration samples:
\begin{enumerate}
\item $W^{(t)} \sim p(W \mid \lambda^{(t-1)}, \kappa^{(t-1)}, \widehat{W}_1)$ --- a truncated normal (Half-normal with observed data)
\item $\lambda^{(t)} \sim p(\lambda \mid W^{(t)})$ --- a gamma distribution
\item $\kappa^{(t)} \sim p(\kappa \mid W^{(t)}, \widehat{W}_1)$ --- a gamma distribution
\end{enumerate}

All conditionals are conjugate and admit closed-form sampling. Unlike the 9-dimensional quantile
approach (v6.0), this requires no matrix inversions or Cholesky decompositions---all operations
are scalar.

\paragraph{Convergence and Cost.}
We run 256 total iterations with 64 burn-in, retaining 192 samples. Mixing is fast due to
the low-dimensional parameter space and conjugate structure. Initialization at $\lambda^{(0)} =
\kappa^{(0)} = 1$ (the prior means) provides a good starting point.

We track $\lambda$ mixing via its coefficient of variation (CV) and effective sample size (ESS).
If $\lambda$ CV $< 0.1$ or $\lambda$ ESS $< 20$, we flag a potential convergence issue in
diagnostics, though this rarely occurs in practice.

Computational cost per posterior update is $O(N_{\text{gibbs}}) \approx 256$ scalar operations---
negligible compared to measurement time ($<1$ms on modern hardware). This is substantially faster
than the 9-dimensional approach (v6.0), which required $O(d^3 + d^2 \cdot N_{\text{gibbs}})$
operations dominated by Cholesky decompositions.

\paragraph{Leak Probability.}
The leak probability is estimated as the Monte Carlo fraction of posterior samples exceeding
the effective threshold:
\begin{equation}
\widehat{P}(W > \theta_{\text{eff}} \mid \widehat{W}_1) = \frac{1}{N_{\text{keep}}} \sum_{s=1}^{N_{\text{keep}}} \mathbf{1}\!\left[W^{(s)} > \theta_{\text{eff}}\right]
\end{equation}

\subsection{Block Bootstrap for Autocorrelation}
\label{app:bootstrap}

\paragraph{Why Standard Bootstrap Fails.}
Timing measurements exhibit autocorrelation: nearby samples are more similar than distant ones due to cache state, CPU frequency scaling, and OS scheduling. Standard bootstrap assumes i.i.d.\ samples, systematically underestimating variance and producing overconfident posteriors.

For example, if consecutive measurements have autocorrelation $\rho = 0.3$, the effective sample size may be 30--50\% lower than the nominal count. Ignoring this inflates the false positive rate---the main failure mode we aim to prevent.

\paragraph{Stream-Based Block Bootstrap.}
We use block bootstrap on the \emph{acquisition stream}---the interleaved sequence of (class, timing)
pairs in measurement order---rather than per-class samples. This preserves the true temporal
dependence structure.

The algorithm resamples contiguous blocks of length $L$ with replacement, reconstructs a bootstrap
stream of the same length, splits by class, and computes $\widehat{W}_1$ for each replicate.
Block resampling may produce slightly unequal class sizes; when this occurs, we use the general
1D Wasserstein formula for unequal sample sizes or truncate the larger class by one sample. Over
$B = 2000$ iterations, we estimate the variance $v_{\text{cal}}$ of $\widehat{W}_1$ under the
calibration sample size.

\paragraph{Block Length Selection.}
We use the Politis-White automatic block length selection algorithm with a class-conditional modification. Standard Politis-White estimates autocorrelation from raw lag-$k$ pairs, but with interleaved sampling, a lag-$k$ pair may cross class boundaries. Instead, we compute autocorrelation using only same-class pairs at each acquisition-stream lag, then take the conservative maximum across classes.

The optimal block length scales as $\hat{b} \propto T^{1/3}$ where $T$ is the stream length. We clamp $\hat{b} \in [10, \min(3\sqrt{T}, T/3)]$ to prevent degenerate blocks. In high-autocorrelation regimes (detected when $\rho_{\text{max}}(10) > 0.3$), we inflate the block length by 50\% for additional conservatism.

\paragraph{Variance Scaling.}
The key insight enabling adaptive sampling is that W$_1$ estimator variance scales as $1/n_{\text{blocks}}$.
We estimate a \textbf{variance-rate} constant during calibration:
\begin{equation}
v_{\text{rate}} = v_{\text{cal}} \cdot n_{\text{blocks,cal}}
\end{equation}

During the adaptive loop with $n$ samples:
\begin{equation}
v_n = \frac{v_{\text{rate}}}{n_{\text{blocks}}(n)}
\end{equation}
where $n_{\text{blocks}}(n) = \max(1, \lfloor n/L \rfloor)$ is the number of approximately independent
blocks. This allows cheap posterior updates without re-bootstrapping at each batch.

The terminology ``$n_{\text{blocks}}$'' replaces the earlier ``$n_{\text{eff}}$'' (effective sample
size) to avoid confusion with IACT-based estimators. The block count provides a conservative lower
bound on the number of approximately independent observations given the dependence structure.

\subsection{Quality Gates}
\label{app:gates}

Quality gates detect when data quality is insufficient for a confident verdict, forcing an \textsc{Inconclusive} outcome rather than an unreliable Pass or Fail. This is the key mechanism ensuring calibrated verdicts.

\paragraph{S1: Insufficient Information Gain.}
We compute the KL divergence between Gaussian surrogates of the prior and posterior
for the 1D W$_1$ distance $\delta$:
\begin{equation}
\mathrm{KL} = \frac{1}{2}\left(
\frac{V_{\text{post}}}{V_0^{\text{marginal}}}
+ \frac{\mu_{\text{post}}^2}{V_0^{\text{marginal}}}
- 1 + \ln\frac{V_0^{\text{marginal}}}{V_{\text{post}}}
\right)
\end{equation}
where $V_{\text{post}}$ is the posterior variance of $\delta$ (estimated from Gibbs samples)
and $V_0^{\text{marginal}} = 2\sigma^2$ is the marginal prior variance (§3.4.3).

The threshold of 0.7 nats was selected empirically from pilot studies on synthetic data with known null effects—values below this consistently indicated insufficient posterior movement to support reliable verdicts. The 0.001 nats threshold for NotLearning similarly reflects the practical lower bound of KL divergence detectability given MCMC approximation error with 192 retained samples.

\textbf{DataTooNoisy}: If KL $< 0.7$ nats after calibration, the posterior has barely moved from the prior---the data is too noisy to distinguish signal from noise.

\textbf{NotLearning}: If the sum of inter-batch KL divergences falls below 0.001 for 5+ consecutive batches, the posterior has stopped updating. Additional samples will not help.

\paragraph{S2: Condition Drift Detected.}
The covariance rate $\Sigma_{\text{rate}}$ is estimated during calibration. If measurement conditions change during the adaptive loop, this estimate becomes invalid. We detect drift by comparing:
\begin{itemize}
\item Variance ratio: $\sigma_{\text{post}}^2 / \sigma_{\text{cal}}^2$ outside $[0.5, 2.0]$
\item Autocorrelation change: $|\rho_{\text{post}}(1) - \rho_{\text{cal}}(1)| > 0.3$
\item Mean drift: $|\mu_{\text{post}} - \mu_{\text{cal}}| / \sigma_{\text{cal}} > 3.0$
\end{itemize}
These thresholds were calibrated on pilot experiments to detect gross non-stationarity (2$\times$ variance change, significant autocorrelation shift, or large mean drift) while tolerating natural measurement variation in realistic CI environments. Any trigger yields \textsc{Inconclusive} with reason \textsc{ConditionsChanged}.

\paragraph{O1: Would Take Too Long.}
We extrapolate time to decision based on current convergence rate. If projected time exceeds 10$\times$ the configured budget, we stop early with \textsc{Inconclusive} rather than waste resources on a test that cannot conclude.

\paragraph{O2: Resource Budget Exceeded.}
If elapsed time exceeds the configured time budget, or total samples exceed the configured maximum, we stop with \textsc{Inconclusive} and report the current posterior probability.

\paragraph{Likelihood Inflation Warning.}
When the posterior mean of $\kappa$ falls below 0.3, the likelihood covariance was effectively scaled up by $>3\times$ for robustness. This indicates potential covariance misestimation and is reported in diagnostics, though it does not block verdicts.

\subsection{Measurement Floor Derivation}
\label{app:floor}

A critical design element is distinguishing what the user \emph{wants} to detect ($\theta_{\text{user}}$) from what the measurement \emph{can} detect ($\theta_{\text{floor}}$).

\paragraph{Floor-Rate Constant.}
During calibration, we estimate the null distribution of $\widehat{W}_1$ under the hypothesis of no
timing leak. This is done by computing $\widehat{W}_1$ between resampled baseline-baseline and
sample-sample splits using the same block-bootstrap procedure that respects temporal dependence.

For each null replicate $\widehat{W}_1^{(b)}$ at calibration sample size $n_{\text{cal}}$:
\begin{equation}
c_{\text{floor}} = q_{0.95}\!\left(\widehat{W}_1^{(b)} \cdot \sqrt{n_{\text{blocks,cal}}}\right)
\end{equation}

This floor-rate constant captures the 95th percentile of the null distribution of raw W$_1$, scaled to
be sample-size-independent. Critically, we use \emph{raw} $\widehat{W}_1$ values (not debiased or
clamped), ensuring the floor accounts for the natural positive bias of W$_1$ under the null hypothesis.

\paragraph{Dynamic Measurement Floor.}
During the adaptive loop with $n$ samples, the statistical floor shrinks as more data is collected:
\begin{equation}
\theta_{\text{floor,stat}}(n) = \frac{c_{\text{floor}}}{\sqrt{n_{\text{blocks}}(n)}}
\end{equation}

The timer also imposes a floor based on quantization:
\begin{equation}
\theta_{\text{tick}} = \frac{\text{1 tick (ns)}}{K}
\end{equation}
where $K$ is the batch size (if batching is used). The combined floor is:
\begin{equation}
\theta_{\text{floor}}(n) = \max\!\left(\theta_{\text{floor,stat}}(n), \theta_{\text{tick}}\right)
\end{equation}

Treating $\theta_{\text{tick}}$ as an irreducible floor is a conservative design choice:
while sub-tick mean shifts are theoretically resolvable with sufficient independent jitter,
we assume worst-case timer behavior for security testing.

\paragraph{Effective Threshold.}
The effective threshold used for inference is:
\begin{equation}
\theta_{\text{eff}} = \max(\theta_{\text{user}}, \theta_{\text{floor}})
\end{equation}

When $\theta_{\text{eff}} > \theta_{\text{user}}$:
\begin{itemize}
\item \textbf{Fail propagates}: Detecting $W > \theta_{\text{eff}}$ implies $W > \theta_{\text{user}}$,
      since $\theta_{\text{eff}} \geq \theta_{\text{user}}$ by construction.
\item \textbf{Pass does not propagate}: ``No effect above $\theta_{\text{eff}}$'' is compatible with
      effects in the gap $(\theta_{\text{user}}, \theta_{\text{eff}}]$. We return \textsc{Inconclusive}
      with reason \textsc{ThresholdElevated}.
\end{itemize}

As sampling continues, $\theta_{\text{floor}}(n)$ decreases. If it drops to $\theta_{\text{user}}$ or below, Pass becomes possible (subject to the posterior). This is why adaptive sampling can eventually reach Pass even when early calibration shows an elevated floor.

\subsection{Adaptive Batching}
\label{app:batching}

On platforms with coarse timer resolution (e.g., Apple Silicon's \ns{42} \texttt{cntvct\_el0}),
individual operation timing can be dominated by quantization noise. \tacet automatically detects
this during a pilot phase and enables adaptive batching: rather than timing single operations,
it times batches of $K$ operations and divides by $K$.

\tacet selects the batch size to achieve at least 50 timer ticks per measurement, clamped to
$K \in [1, 20]$ to avoid microarchitectural artifacts. When $K = 20$ is insufficient to reach
50 ticks, residual quantization is absorbed into $\theta_{\text{tick}}$ (\Cref{app:floor}),
raising the measurement floor and increasing the likelihood of an Inconclusive verdict. Batching
is automatically disabled on x86\_64 (where \texttt{rdtsc} provides sufficient resolution) or
when PMU timers are used.
