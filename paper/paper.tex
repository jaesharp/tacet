\begin{abstract}
%-------------------------------------------------------------------------------
% ~150-200 words summarizing the key contributions
% Hook: existing tools give false confidence
% Solution: calibrated Bayesian verdicts
% Results: highlight key evaluation findings

Timing side-channel attacks remain a persistent threat to cryptographic implementations.
Existing detection tools like \dudect and \tlsfuzzer provide binary pass/fail verdicts,
but offer no indication when measurement noise prevents reliable conclusions---leading
developers to place false confidence in potentially vulnerable code.
%
We present \tacet, a timing side-channel detection library that provides
\emph{calibrated verdicts}: when evidence is insufficient, \tacet reports
Inconclusive rather than guessing, indicating when evidence is insufficient.
%
\tacet achieves this through Bayesian inference over quantile differences,
outputting interpretable posterior probabilities $\prob{\text{leak} > \theta \given \text{data}}$
rather than p-values.
Key innovations include block bootstrap covariance estimation to handle
autocorrelated measurements, quality gates that detect when the model
cannot reliably distinguish signal from noise, and explicit measurement
floor tracking.
%
Our evaluation on synthetic benchmarks and real cryptographic libraries demonstrates
that \tacet maintains calibrated verdicts across varying noise levels and
autocorrelation conditions where existing tools produce unreliable results.
\tacet is implemented in Rust with C/C++, Go, and JavaScript bindings, designed for
integration into continuous integration pipelines.

\end{abstract}

\section{Introduction}
\label{sec:intro}

Timing side-channel detection is a critical part of validating cryptographic
implementations, yet existing tools provide binary pass/fail verdicts without
indicating when measurement conditions prevent reliable conclusions. When noise
or autocorrelation degrades data quality, these tools may output ``pass'' when
evidence is genuinely insufficient---a problem we term \emph{false confidence}.
A developer who sees a passing result naturally assumes their code is safe,
unaware that the test lacked the statistical power to detect a real vulnerability.

Timing side-channels remain a serious threat to deployed cryptographic systems.
Recent attacks have demonstrated practical exploitation across diverse settings:
Lucky~Thirteen~\cite{lucky13} exploited CBC padding timing in TLS, requiring only
a few thousand queries; the Raccoon Attack~\cite{raccoon} recovered DH secrets
via microsecond-level timing differences; and the MARVIN attack~\cite{marvin}
showed that RSA PKCS\#1 v1.5 implementations in major libraries leak plaintext
through timing. These attacks span network-accessible APIs (where Timeless Timing
Attacks~\cite{timeless} enable LAN-like precision via request multiplexing) to
co-located adversaries in SGX enclaves and containerized environments.

The standard approach to timing validation uses statistical tests that compare
execution time distributions between fixed and random inputs. \dudect~\cite{dudect},
the most widely adopted tool, applies Welch's t-test and reports a binary verdict.
More recent tools like \rtlf~\cite{rtlf} and \silent~\cite{silent} use quantile
comparisons with bootstrap-based inference. However, all existing tools share
a fundamental limitation: they output binary verdicts (leak detected / no leak
detected) without distinguishing between ``the implementation is safe'' and
``we couldn't tell.'' Under noisy conditions---common in CI environments with
shared resources and variable system load---this conflation leads to unreliable
results: false positives that waste developer time, and worse, false negatives
that provide unwarranted confidence in vulnerable code.

We present \tacet, a timing side-channel detection tool that provides
\emph{calibrated verdicts}. Rather than forcing a binary decision, \tacet
outputs three possible verdicts: \textsc{Pass} (high confidence of no leak),
\textsc{Fail} (high confidence of leak), or \textsc{Inconclusive} (insufficient
evidence). This calibration is achieved through Bayesian inference: \tacet
computes the posterior probability $\prob{\text{leak} > \theta \given \text{data}}$
and only emits Pass or Fail when this probability crosses a decision threshold
(default 5\% or 95\%). When evidence is insufficient, quality gates detect this
condition and return Inconclusive rather than guessing.

\tacet adopts the quantile-difference test statistic from \silent~\cite{silent},
which compares timing distributions via their deciles rather than means. This
captures both uniform timing shifts (from branch differences) and tail effects
(from cache behavior). Our contribution is not the test statistic itself, but
the probabilistic framework that enables calibrated uncertainty quantification:
a robust Bayesian model, quality gates that detect degraded measurement conditions,
and an explicit measurement floor that prevents claiming ``no leak'' when the
system cannot resolve effects at the requested threshold.

\section{Background and Related Work}
\label{sec:background}

%-------------------------------------------------------------------------------
\subsection{Timing Side-Channel Attacks}
\label{sec:background:attacks}

Timing side-channels arise when a cryptographic implementation's execution time
depends on secret data. An attacker who can measure this timing---either directly
or through statistical analysis of many queries---may recover secret keys or
plaintext. The threat model varies by attacker capability:

\paragraph{Remote network attackers.}
Traditional timing attacks over the internet face millisecond-level jitter,
limiting exploitability. However, Van~Goethem~\etal~\cite{timeless} demonstrated
that HTTP/2 request multiplexing enables \emph{timeless timing attacks}, where
relative timing between concurrent requests eliminates network jitter. This
effectively gives remote attackers LAN-like precision ($\sim$100\,ns), making
previously unexploitable leaks dangerous.

\paragraph{Adjacent network attackers.}
Attackers on the same LAN or in the same data center can achieve microsecond
precision. The Raccoon Attack~\cite{raccoon} exploited 1--10\,$\mu$s timing
differences in TLS-DH implementations to recover pre-master secrets.

\paragraph{Co-located attackers.}
In shared environments (cloud VMs, containers, SGX enclaves), attackers may
achieve cycle-level timing resolution through cache timing, hyperthreading
interference, or direct hardware access. KyberSlash~\cite{kyberslash} demonstrated
that $\sim$20-cycle timing leaks in post-quantum implementations can be exploited
by co-located adversaries.

%-------------------------------------------------------------------------------
\subsection{Existing Detection Tools}
\label{sec:background:tools}

Several tools exist for detecting timing side-channels during development.

\paragraph{\dudect~\cite{dudect}.}
The most widely used tool, \dudect applies Welch's t-test to compare timing
distributions between fixed and random inputs. It is simple, fast, and available
in C with Rust bindings. However, it compares only means (missing tail effects),
assumes normality (violated by timing data), outputs only a t-statistic (not
interpretable as a probability), and provides no indication when measurement
noise makes the result unreliable.

\paragraph{\tlsfuzzer~\cite{tlsfuzzer}.}
A network-level timing analysis tool that applies multiple statistical tests
(Wilcoxon, Friedman, etc.) to TLS implementations. It handles some forms of
autocorrelation and provides bounded Type-I error, but requires complex setup
for protocol fuzzing and outputs binary verdicts without uncertainty quantification.

\paragraph{\rtlf~\cite{rtlf}.}
Introduced at USENIX Security 2024, \rtlf compares nine deciles between timing
distributions using an i.i.d.\ bootstrap with 10,000 iterations. This captures
distributional differences beyond means. However, the i.i.d.\ bootstrap assumes
independent samples---violated when timing measurements exhibit autocorrelation
from cache state or CPU frequency scaling. \rtlf outputs binary verdicts and
operates only on pre-collected measurement files (no integrated measurement).

\paragraph{\silent~\cite{silent}.}
The most statistically sophisticated prior work, \silent uses a quantile-based
test statistic with $m$-dependent block bootstrap that correctly handles
autocorrelated data. It provides bounded Type-I error via ``relevance testing''
against a user-defined threshold $\Delta$. However, \silent outputs binary
verdicts only: when evidence is marginal, it must choose Pass or Fail rather
than reporting uncertainty. The paper explicitly acknowledges stationarity as
a limitation but does not actively detect violations.

\section{Tacet's Approach}
\label{sec:approach}

\tacet is designed around a core principle: when measurement conditions cannot
support a confident conclusion, the tool should say so rather than guess. This
section describes the statistical methodology that enables calibrated verdicts.

\paragraph{Threshold selection.}
The threshold $\theta$ defines the minimum effect size worth detecting. There is no
single correct value---it depends on the threat model. We provide presets based on
published attacks: \textsc{SharedHardware} (0.4\,ns, $\approx$2 cycles at 5\,GHz)
for co-resident attackers; \textsc{AdjacentNetwork} (100\,ns) for LAN attackers
leveraging HTTP/2 multiplexing~\cite{timeless}; \textsc{RemoteNetwork} (50\,$\mu$s)
for general internet exposure. Users may specify custom thresholds.

%-------------------------------------------------------------------------------
\subsection{Calibrated Verdicts}
\label{sec:approach:decisions}

Unlike binary pass/fail tools, \tacet outputs three possible verdicts:

\begin{itemize}
    \item \textbf{Pass}: The posterior probability of a timing leak exceeding
          the threshold is below 5\%: $\prob{\text{leak} > \theta \given \text{data}} < 0.05$.
          High confidence that no exploitable leak exists.

    \item \textbf{Fail}: The posterior probability exceeds 95\%:
          $\prob{\text{leak} > \theta \given \text{data}} > 0.95$.
          High confidence that a timing leak exists.

    \item \textbf{Inconclusive}: The posterior probability falls between the
          decision thresholds, \emph{or} a quality gate has detected that
          measurement conditions cannot support a reliable verdict.
\end{itemize}

This three-way structure resolves a fundamental problem with binary tools:
they conflate ``the implementation is safe'' with ``we couldn't tell.'' When
\dudect reports ``Pass'' under noisy conditions, this may reflect genuine
constant-time behavior or simply insufficient statistical power. \tacet makes
this distinction explicit---Inconclusive means the test should be re-run under
better conditions or with more samples, not that the code is safe.

For CI integration, this calibration prevents flaky tests: rather than
randomly passing or failing based on environmental noise, \tacet returns
Inconclusive when conditions are poor. Developers can configure how to handle
Inconclusive results (retry, warn, or fail-closed) based on their security requirements.

%-------------------------------------------------------------------------------
\subsection{Quantile-Based Test Statistic}
\label{sec:approach:quantile}

We adopt the quantile-difference test statistic and max-$k$ decision rule from
\silent~\cite{silent}, embedding them in a probabilistic framework that enables
calibrated uncertainty quantification. Our contribution is not the test statistic
itself, but the Bayesian inference framework that produces calibrated verdicts
with interpretable posterior probabilities.

Rather than comparing means (as in \dudect's t-test), we compare timing
distributions via their deciles. For fixed-input samples $F$ and random-input
samples $R$, we compute the vector of quantile differences:
\begin{equation}
    \Delta = \hat{q}(F) - \hat{q}(R) \in \mathbb{R}^9
\end{equation}
where $\hat{q}_p$ denotes the empirical $p$-th quantile for $p \in \{0.1, 0.2, \ldots, 0.9\}$.

The decision functional is $m(\delta) = \max_k |\delta_k|$---a timing leak is
declared when the maximum absolute quantile difference exceeds the threshold $\theta$.

\paragraph{Why quantiles?}
Timing leaks manifest in different patterns. A branch that depends on secret
data causes a \emph{uniform shift}: all quantiles move by approximately the
same amount. A cache-timing leak causes a \emph{tail effect}: only some operations
experience cache misses, shifting upper quantiles while leaving lower quantiles
unchanged. A mean-comparison test like \dudect would detect uniform shifts but
may miss tail effects. Quantile differences capture both patterns.

%-------------------------------------------------------------------------------
\subsection{Bayesian Inference}
\label{sec:approach:inference}

\tacet uses Bayesian inference to compute posterior probabilities rather than
p-values. This provides several advantages:

\paragraph{Interpretable output.}
When \tacet reports ``72\% probability of a leak,'' this means exactly what it
says: given the observed data and our model, 72\% of the posterior probability
mass corresponds to effects exceeding the threshold. Contrast this with a p-value,
which answers a different question (``if there were no leak, how likely is data
this extreme?'') and is routinely misinterpreted.

\paragraph{Valid adaptive sampling.}
Frequentist methods suffer from \emph{optional stopping}: if you keep sampling
until you get a significant result, you inflate your false positive rate.
Bayesian inference does not have this problem---the posterior probability is
valid regardless of when you stop, a consequence of the likelihood principle.
This enables \tacet's adaptive sampling: collect data until confident, then stop.

\paragraph{Principled handling of prior information.}
The prior encodes what we know before seeing data. \tacet calibrates the prior
so that it expresses genuine uncertainty: neither strongly favoring ``leak''
nor ``no leak.'' This prevents the prior from dominating when data is limited
while allowing confident conclusions when evidence is strong.

The full model specification (likelihood, prior, Gibbs sampler) is provided in
Appendix~\ref{app:methodology}. At a high level: we model the observed quantile
differences with a robust Student-$t$ likelihood that automatically inflates
uncertainty when the covariance estimate is too small, and place a Student-$t$
prior on the true effect that adapts its scale based on the threshold. The
posterior is approximated via MCMC, with $<$1\,ms computational overhead.

%-------------------------------------------------------------------------------
\subsection{Handling Autocorrelation}
\label{sec:approach:autocorrelation}

Timing measurements exhibit \emph{autocorrelation}: nearby samples are more
similar than distant ones due to cache state, CPU frequency scaling, and OS
scheduling. Standard statistical methods assume independent samples; violating
this assumption causes underestimated variance and inflated false positive rates.

\tacet addresses this through block bootstrap on the \emph{acquisition stream}---the
interleaved sequence of (class, timing) pairs in measurement order. Rather than
resampling individual measurements (which would destroy the dependence structure),
we resample contiguous blocks and reconstruct timing distributions from each
bootstrap replicate. This preserves temporal dependence within blocks while
estimating the variance inflation caused by autocorrelation.

Block length is selected automatically using the Politis-White algorithm~\cite{politiswhite},
which estimates the optimal block size from the autocorrelation structure. We
also compute the \emph{integrated autocorrelation time} (IACT), which quantifies
how many nominal samples equal one independent observation. The effective sample
size $n_{\text{eff}} = n / \hat{\tau}$ may be substantially smaller than the
nominal count when autocorrelation is strong.

\silent also uses block bootstrap with $m$-dependence, so autocorrelation handling
per se is not a unique differentiator. The key differences are: (1) \tacet does
not assume a specific parametric dependence structure; (2) \tacet actively monitors
for stationarity violations via quality gates; (3) \tacet outputs Inconclusive
when autocorrelation is too severe to handle, rather than forcing a verdict.

%-------------------------------------------------------------------------------
\subsection{Quality Gates}
\label{sec:approach:gates}

Quality gates are the mechanism that enables calibrated verdicts. They detect
when measurement conditions are too poor to support a confident conclusion,
triggering Inconclusive rather than an unreliable Pass or Fail.

\paragraph{Gate 1: Insufficient information gain.}
After calibration, \tacet computes the KL divergence between the prior and
posterior distributions. If KL $<$ 0.7 nats, the posterior has barely moved
from the prior---the data is too noisy to distinguish signal from noise.
This triggers Inconclusive with reason \textsc{DataTooNoisy}.

During adaptive sampling, if the KL divergence between successive posteriors
collapses (total $<$ 0.001 nats over 5+ batches), the model has stopped
learning from new data. More samples will not help. This triggers Inconclusive
with reason \textsc{NotLearning}.

\paragraph{Gate 2: Would take too long.}
\tacet extrapolates time to decision based on the current convergence rate. If
projected time exceeds 10$\times$ the configured budget, it stops early with
Inconclusive rather than wasting resources on a test that cannot conclude within
practical limits.

\paragraph{Gate 3: Resource budget exceeded.}
If elapsed time exceeds the configured time budget or samples exceed the maximum,
\tacet reports Inconclusive with the current posterior probability, allowing the
user to interpret the partial result.

\paragraph{Gate 4: Condition drift detected.}
The covariance estimate from calibration assumes stationary conditions. If
measurement statistics change during the adaptive loop (variance ratio outside
$[0.5, 2.0]$, lag-1 autocorrelation change $>0.3$, or mean drift $>3\sigma$),
this indicates non-stationarity. \tacet reports Inconclusive with reason
\textsc{ConditionsChanged} rather than producing a verdict based on invalid assumptions.

\silent acknowledges stationarity as a limitation but does not actively detect
violations. \tacet's quality gates fill this gap: when conditions change during
measurement, \tacet reports Inconclusive rather than producing a potentially
misleading verdict.

%-------------------------------------------------------------------------------
\subsection{Measurement Floor}
\label{sec:approach:floor}

A subtle failure mode of timing analysis is claiming ``no leak'' when the
measurement system cannot actually resolve effects at the requested threshold.
\tacet makes this explicit through the \emph{measurement floor} $\theta_{\text{floor}}$.

The measurement floor is the smallest effect the system can reliably detect given
current conditions. It has two components: (1) a statistical component that
shrinks as $\theta_{\text{floor,stat}}(n) = c_{\text{floor}} / \sqrt{n_{\text{eff}}}$,
where $c_{\text{floor}}$ is calibrated from the noise structure; and (2) a timer
resolution component $\theta_{\text{tick}}$ based on quantization.

The \emph{effective threshold} used for inference is:
\begin{equation}
    \theta_{\text{eff}} = \max(\theta_{\text{user}}, \theta_{\text{floor}})
\end{equation}

When $\theta_{\text{eff}} > \theta_{\text{user}}$:
\begin{itemize}
    \item \textbf{Fail still propagates}: Detecting an effect above $\theta_{\text{eff}}$
          implies an effect above $\theta_{\text{user}}$ (since $\theta_{\text{eff}} \geq \theta_{\text{user}}$).
    \item \textbf{Pass does not propagate}: ``No effect above $\theta_{\text{eff}}$'' is
          compatible with effects in the gap $(\theta_{\text{user}}, \theta_{\text{eff}}]$.
          \tacet returns Inconclusive with reason \textsc{ThresholdElevated}.
\end{itemize}

This prevents false confidence: \tacet never claims ``safe at 100\,ns'' when
it can only resolve effects down to 500\,ns. As sampling continues, the floor
decreases; if it drops to $\theta_{\text{user}}$ or below, Pass becomes possible.

\section{Implementation}
\label{sec:implementation}

\tacet is implemented in Rust, with the statistical core comprising approximately
% TODO: count actual KLOC
5,000 lines of code. The block bootstrap is parallelized using Rayon, providing
4--8$\times$ speedup on multi-core systems. Foreign function interfaces expose
the same API surface to C, C++, Go, and JavaScript (via WebAssembly with native
timer bindings for Node.js, Deno, and Bun).

%-------------------------------------------------------------------------------
\subsection{Measurement Infrastructure}
\label{sec:implementation:measurement}

High-resolution timing is platform-specific. On x86\_64, we use the \texttt{rdtsc}
instruction, which provides cycle-level resolution. On ARM64, we use the
\texttt{cntvct\_el0} counter, which has coarser resolution (typically 24\,MHz
on Apple Silicon, corresponding to $\sim$42\,ns ticks).

For scenarios requiring cycle-level precision on ARM64, \tacet optionally supports
Performance Monitoring Unit (PMU) counters: \texttt{kperf} on macOS (requires
elevated privileges) and \texttt{perf\_event} on Linux (requires \texttt{CAP\_PERFMON}
or root). These provide sub-nanosecond resolution but are opt-in due to privilege
requirements.

Measurements are collected as an interleaved acquisition stream: fixed and random
inputs are alternated to minimize systematic bias from CPU state changes. All
inputs are pre-generated before the measurement loop to avoid timing contamination
from random number generation.

%-------------------------------------------------------------------------------
\subsection{Adaptive Batching}
\label{sec:implementation:batching}

On platforms with coarse timer resolution, individual operation timing may be
dominated by quantization noise. \tacet automatically detects this condition
during a pilot phase and enables \emph{adaptive batching}: rather than timing
single operations, it times batches of $K$ operations and divides by $K$.

The batch size is selected to achieve at least 50 timer ticks per measurement,
clamped to $K \in [1, 20]$ to avoid microarchitectural artifacts from excessive
repetition. Batching is automatically disabled on x86\_64 (where \texttt{rdtsc}
provides sufficient resolution) or when PMU timers are used.

%-------------------------------------------------------------------------------
\subsection{CI Integration}
\label{sec:implementation:bindings}

\tacet is designed for continuous integration pipelines. Key properties:

\begin{itemize}
    \item \textbf{Non-flaky}: The three-way verdict system means tests return
          Inconclusive under poor conditions rather than randomly passing or
          failing. This eliminates the ``rerun until green'' problem.

    \item \textbf{Configurable budgets}: Time and sample budgets can be set per
          test, allowing quick checks during development (10\,s) and thorough
          validation before release (60\,s+).

    \item \textbf{Exit codes}: The CLI returns distinct exit codes for Pass (0),
          Fail (1), and Inconclusive (2), enabling CI scripts to handle each
          case appropriately. Organizations can configure fail-closed (treat
          Inconclusive as failure) or fail-open (treat as warning) policies.

    \item \textbf{Structured output}: JSON output includes the posterior probability,
          effect size estimate with credible intervals, and diagnostic information,
          enabling automated analysis and trend tracking.
\end{itemize}

\section{Evaluation}
\label{sec:evaluation}

We evaluate \tacet on three questions: (1) Does it provide calibrated verdicts
under varying measurement conditions? (2) How does it compare to existing tools?
(3) Does it work on real cryptographic implementations?

%-------------------------------------------------------------------------------
\subsection{Methodology}
\label{sec:evaluation:methodology}

\paragraph{Synthetic benchmarks.}
Synthetic experiments use generated timing distributions with controlled effect
sizes, autocorrelation structures, and noise levels. This allows precise ground-truth
evaluation independent of hardware. We inject effects as shifts in the mean timing
(simulating branch differences) or shifts in upper quantiles only (simulating
cache effects), with autocorrelation modeled as AR(1) processes at varying
correlation coefficients $\rho \in \{-0.6, -0.3, 0, 0.3, 0.6\}$.

\paragraph{Cryptographic library tests.}
Real-world tests run on AWS EC2 instances: \texttt{c8g.4xlarge} (ARM64 Graviton3,
16 vCPUs) and \texttt{c8a.4xlarge} (x86\_64 AMD EPYC, 16 vCPUs). These represent
typical CI infrastructure. We test against cryptographic implementations from
multiple Rust libraries (RustCrypto, ring, dalek, pqcrypto) covering symmetric,
asymmetric, hash, AEAD, and post-quantum algorithms.

\paragraph{Tools compared.}
We compare \tacet against \dudect, \silent, \rtlf, and several additional
statistical tests (Kolmogorov-Smirnov, Anderson-Darling, TVLA t-test, MONA).
For fair comparison, all tools receive identical sample counts (10,000 samples
per class for synthetic benchmarks).

\paragraph{Metrics.}
We report: (1) \emph{False positive rate} (FPR): proportion of null-effect
trials yielding Fail; (2) \emph{Detection rate}: proportion of effect-present
trials yielding Fail; (3) \emph{Inconclusive rate}: proportion of trials yielding
Inconclusive (applicable only to \tacet). For tools without an Inconclusive
category, all verdicts are either Pass or Fail.

%-------------------------------------------------------------------------------
\subsection{Synthetic Benchmarks}

\paragraph{False positive rate under autocorrelation.}
Figure~\ref{fig:fpr_heatmap} shows verdict distributions under the null hypothesis
(no injected effect) across varying autocorrelation structures. Each cell shows
the proportion of trials yielding each verdict type.

Under i.i.d.\ conditions ($\rho = 0$), most tools correctly return ``no effect
found'' with low false positive rates. However, as positive autocorrelation
increases ($\rho = 0.3, 0.6$), tools that assume independence show elevated
false positive rates: the i.i.d.\ bootstrap underestimates variance, causing
tests to reject the null hypothesis when they should not.

\tacet shows a distinctive pattern: under high autocorrelation, it returns
Inconclusive (orange) rather than incorrect Pass or Fail verdicts. This is the
calibration property in action---when the effective sample size drops due to
autocorrelation, \tacet's quality gates detect that the posterior hasn't moved
sufficiently from the prior and refuse to commit to a verdict.

\silent also handles autocorrelation correctly via $m$-dependent bootstrap, showing
consistently low false positive rates. The key difference is that \silent must
still output Pass or Fail, while \tacet can report uncertainty explicitly.

effect size increases, the posterior shifts toward Fail, eventually crossing
the 95\% threshold. This gradual transition reflects the underlying probability
rather than a sharp binary cutoff.

\subsection{Results}
\label{sec:evaluation:results}

\paragraph{Calibration validation.}
We validated \tacet's probabilistic guarantees using the calibration test suite.
Running 500 trials per attacker model under the null hypothesis (random-vs-random,
no injected effect), we observed 0 false positives across all models
(Wilson 95\% CI: [0\%, 0.76\%]), well within our $\leq$5\% target.
%
For detection power, at 1$\times$ threshold (100\,ns for \textsc{AdjacentNetwork}),
\tacet achieved 100\% detection across 200 trials (95\% CI: [98.1\%, 100\%]).
At 0.5$\times$ threshold (50\,ns), detection was 0\% as expected---effects below
threshold should not trigger Fail verdicts.
%
Effect size estimates showed low bias: $<$4\% at the threshold and $<$2\% for
effects $\geq$2$\times$ threshold, confirming accurate quantification.

\paragraph{Zero false positives on constant-time implementations.}
Across all 48 tests on known constant-time implementations, \tacet returned
zero false positives (Pass or Inconclusive, never Fail on safe code). This
validates the calibration: \tacet correctly identifies implementations that
exhibit no exploitable timing differences.

\paragraph{Accidental MARVIN discovery.}
While building our test corpus of ``known safe'' implementations, we \emph{accidentally}
rediscovered the MARVIN attack (CVE-2023-49092)~\cite{marvin} in the RustCrypto
\texttt{rsa} crate (v0.9.9). \tacet detected a $\sim$250\,ns timing difference
in RSA-1024 decryption, exceeding the \textsc{AdjacentNetwork} threshold (100\,ns).

This was not an intentional attempt to replicate a known vulnerability---we were
unaware of the issue when adding RSA to our test suite. \tacet identified the
leak from first principles: the timing difference originated from data-dependent
branching in big-integer arithmetic during modular exponentiation.

This accidental discovery demonstrates \tacet's practical utility: the tool can
detect real vulnerabilities without prior knowledge of implementation details,
making it valuable for routine validation of cryptographic code.

\paragraph{Post-quantum considerations.}
Post-quantum tests use the \textsc{PostQuantumSentinel} threshold (2\,ns,
$\sim$10 cycles at 5\,GHz), reflecting the sensitivity demonstrated by
KyberSlash-class attacks~\cite{kyberslash}. ML-KEM (Kyber-768) and ML-DSA
(Dilithium3) encapsulation/signing passed all tests. Falcon-512 signing
exhibits intentional timing variation from rejection sampling (the message
is public, and rejection is independent of the secret key), which we document
as expected behavior rather than a vulnerability.

\subsection{Performance}
\label{sec:evaluation:performance}

\tacet's computational overhead is dominated by the block bootstrap during
calibration and the Gibbs sampler during posterior updates.

% TODO: Include actual timing measurements
\paragraph{Inference overhead.}
The statistical inference pipeline adds approximately 100\,ms per analysis,
comprising: block bootstrap covariance estimation ($\sim$50\,ms with 2,000
iterations parallelized across 8 cores), Gibbs sampler ($<$1\,ms for 256
iterations), and quality gate checks ($\sim$5\,ms). This is slower than
\dudect (which computes only a t-statistic) but faster than \rtlf (which
runs 10,000 bootstrap iterations in single-threaded R).

\paragraph{Practical impact.}
For CI integration, the inference overhead is typically negligible compared
to measurement time. A test collecting 50,000 samples of a cryptographic
operation at 1\,$\mu$s/operation spends $\sim$100\,ms on measurement and
$\sim$100\,ms on inference---a 2$\times$ overhead that is acceptable for
pre-deployment validation. For development iteration, shorter time budgets
(10\,s) provide rapid feedback.

\section{Discussion and Limitations}
\label{sec:discussion}

%-------------------------------------------------------------------------------
\subsection{Tradeoffs}
\label{sec:discussion:tradeoffs}

\paragraph{Methodological complexity.}
\tacet's statistical methodology is substantially more complex than a t-test.
The full specification (provided as an artifact) runs to approximately 30 pages,
covering the Bayesian model, bootstrap procedures, quality gates, and measurement
floor calculations. This complexity is the price of calibration: simpler methods
cannot provide the same guarantees about when results are reliable.

For practitioners, this complexity is encapsulated in the library---users need
not understand the statistical details to use \tacet effectively. The API is
intentionally simple: specify a threshold (or attacker model), run the test,
interpret the three-way verdict.

\paragraph{Lower power under noise.}
\tacet is deliberately conservative. When measurement conditions are marginal,
it returns Inconclusive rather than committing to Pass or Fail. This may require
more samples or better conditions to reach a definitive verdict compared to
tools that always produce binary output.

We view this as a feature rather than a limitation. The Inconclusive rate is
the ``price of honesty''---it reflects genuine uncertainty rather than hiding
it behind a confident-sounding verdict. A tool that reports Pass under noisy
conditions is not more powerful; it is overconfident.

\paragraph{Overhead.}
The $\sim$100\,ms inference overhead is acceptable for CI validation but may be
prohibitive for extremely fast feedback loops (e.g., running on every file save).
For development iteration, we recommend using shorter time budgets (10\,s) that
sacrifice some precision for speed.

%-------------------------------------------------------------------------------
\subsection{Limitations}
\label{sec:discussion:limitations}

\paragraph{Stationarity assumption.}
Like all timing analysis tools, \tacet assumes measurements are drawn from a
stationary process (the distribution does not change over time). While quality
gates detect gross violations, subtle drift may go undetected. Long-running
tests should be periodically restarted to refresh the calibration baseline.

\paragraph{Model specification.}
The Bayesian model makes specific choices (Student-$t$ distributions, nine
deciles, fixed degrees of freedom) that were validated on our benchmark suite.
Different application domains may benefit from alternative specifications.
The modular design allows researchers to experiment with variants.

%-------------------------------------------------------------------------------
\subsection{Future Work}
\label{sec:discussion:future}

\tacet includes experimental support for power and electromagnetic side-channel
analysis, using the same statistical methodology with different measurement
infrastructure (oscilloscope traces rather than CPU timers). The feature
extraction and alignment challenges differ, but the core inference framework
applies directly. We plan to validate this capability against standard SCA
benchmarks in future work.

\section{Conclusion}
\label{sec:conclusion}

We have presented \tacet, a timing side-channel detection tool that provides
\emph{calibrated verdicts}. Unlike existing tools that output binary pass/fail
decisions, \tacet explicitly reports when evidence is insufficient, returning
Inconclusive rather than guessing. This calibration is achieved through Bayesian
inference over quantile differences, outputting interpretable posterior probabilities
rather than p-values.

Our evaluation demonstrates that \tacet maintains calibrated verdicts across
varying noise levels and autocorrelation conditions where existing tools produce
unreliable results. On a test suite of 48 cryptographic implementations, \tacet
showed zero false positives while successfully detecting a real vulnerability
(MARVIN/CVE-2023-49092) that was discovered accidentally during corpus construction.

The key innovations enabling calibrated verdicts are: (1) quality gates that
detect when the posterior has not sufficiently separated from the prior; (2) a
robust likelihood that automatically inflates uncertainty when covariance is
underestimated; (3) explicit measurement floor tracking that prevents claiming
``safe'' when resolution is insufficient; and (4) block bootstrap on the acquisition
stream to handle autocorrelated measurements.

\tacet is available as open-source software under the MPL~v2 license, with native
support for Rust, C, C++, Go, and JavaScript. It is designed for integration into
continuous integration pipelines, where the three-way verdict system eliminates
flaky tests by returning Inconclusive rather than randomly passing or failing
under variable conditions.

We hope \tacet will help cryptographic library maintainers validate their
implementations with appropriate confidence: not the false confidence of a
binary tool that always produces an answer, but the calibrated confidence of
a tool that knows when to say ``I don't know.''

\section*{Ethical Considerations}
\label{app:ethics}

\noindent\textbf{Stakeholders.}
We identify three primary stakeholders:
(1)~cryptographic library maintainers, who benefit from improved vulnerability detection;
(2)~end users of cryptographic software, who benefit from more secure implementations; and
(3)~potential attackers, who could misuse timing analysis techniques.

\smallskip
\noindent\textbf{Impacts.}
Following the Menlo Report principles:
\begin{itemize}[nosep,leftmargin=*]
    \item \emph{Beneficence}: \tacet enables proactive detection of timing
          side-channels before deployment, reducing vulnerability windows.
    \item \emph{Respect for Persons}: Our work does not involve human subjects
          or personal data.
    \item \emph{Justice}: By releasing \tacet as open-source (MPL v2), we ensure
          equal access across organizations of all sizes.
    \item \emph{Respect for Law and Public Interest}: Timing side-channel
          detection is an established defensive practice.
\end{itemize}

\smallskip
\noindent\textbf{Potential Harms.}
The statistical techniques underlying \tacet could theoretically help attackers
confirm timing leaks. However, simpler tools (\eg \dudect) already provide
this capability; \tacet's contribution is primarily in reducing false positives
and providing calibrated uncertainty---properties more valuable to defenders.

\smallskip
\noindent\textbf{Decision.}
Publication serves the public interest: \tacet's practical CI integration
lowers the barrier for maintainers to adopt rigorous timing analysis.
